### Context
* 로컬 환경에서 구축한 GitHub 자동화 워크플로우를 운영 서버에 배포하는 과정에서 자격 증명(Credentials) 누락 문제가 발생함.
* n8n 워크플로우를 JSON 형식으로 내보내기(Export)하여 서버에 가져올(Import) 때, 보안 규칙에 의해 노드에 연결된 자격 증명 정보는 워크플로우 파일에 포함되지 않음.
* 이로 인해 서버 환경에서 GitHub API 등 외부 서비스와 통신할 수 없는 현상이 발생하였으며, 이를 해결하기 위한 안전한 자격 증명 마이그레이션(Migration) 방법이 필요함.

### Core
* n8n은 워크플로우 구조와 자격 증명 데이터를 완전히 분리하여 데이터베이스에 관리함. 따라서 CLI(Command Line Interface)를 활용하여 별도로 자격 증명을 내보내고 가져와야 함.
* 로컬 환경에서 자격 증명을 복호화(Decrypted)하여 내보낸 뒤, 운영 서버로 안전하게 옮겨 복원하는 로직은 다음과 같음.

```bash
# 로컬 환경에서 자격 증명 내보내기 (복호화된 상태로 JSON 파일 생성)
npx n8n export:credentials --all --decrypted --file=credentials-backup.json

# 생성된 파일을 서버로 안전하게 전송 (scp 또는 sftp 사용 권장)
scp credentials-backup.json user@server-ip:/path/to/n8n/

# 운영 서버 환경에서 자격 증명 가져오기
npx n8n import:credentials --file=credentials-backup.json

# Docker 환경에서 운영 중인 경우의 가져오기 명령어 예시
docker exec -it n8n-container n8n import:credentials --file=/path/to/credentials-backup.json
```

* **보안 주의사항**: `--decrypted` 플래그 사용 시 비밀번호나 API 토큰이 평문으로 파일에 저장되므로, 마이그레이션 작업이 완료되면 양쪽 서버에서 즉시 `credentials-backup.json` 백업 파일을 삭제해야 함.

### Insight
* n8n이 워크플로우와 자격 증명을 분리하는 아키텍처는 운영 보안(Security)을 높이는 훌륭한 설계다. 워크플로우 파일이 유출되더라도 실제 권한 탈취로 이어지지 않게 방어해주기 때문이다.
* 서버 배포(Deployment) 시 단순히 워크플로우 JSON만 복제하는 것이 아니라, 의존성인 자격 증명의 참조 ID 매핑 과정까지 완벽히 이전해야 정상적인 워크플로우 구동이 가능함을 확인했다.
* CLI를 활용한 수동 마이그레이션은 일회성 작업에는 적합하지만 빈번한 배포에서는 병목이 될 수 있다. 장기적인 관점에서는 환경 변수(Environment Variables)를 활용하여 동적으로 시크릿을 주입하거나, 외부 시크릿 볼트(External Secrets Vault, 예: AWS Secrets Manager, HashiCorp Vault) 연동 기능을 도입하여 안전하고 자동화된 CI/CD 파이프라인을 구축하는 것을 권장한다.
